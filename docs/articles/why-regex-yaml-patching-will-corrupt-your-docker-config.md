# Why Regex YAML Patching Will Corrupt Your Docker Config

> If you're using `sed`, `awk`, or regex to modify `docker-compose.yml` in automation, you will eventually ship a broken config. Here's why, and what to do instead.

**Published:** 2026-03-29 · **Source:** [BUG-125](https://github.com/clawhq/clawhq) — doctor auto-fix replaced regex YAML patching with a proper parser

---

## The Problem

ClawHQ's `doctor --fix` command auto-remediates security issues in running OpenClaw deployments. Three of those fixes modify `docker-compose.yml`: adding `cap_drop: ALL`, injecting `no-new-privileges` into `security_opt`, and setting the container user to `1000:1000`.

The original implementation used regex to patch these values. It worked on our test fixtures. It worked on the default compose file generated by `clawhq build`. Then it didn't.

The failure modes:

1. **Indentation mismatch.** YAML is whitespace-sensitive. A regex that matches `cap_drop:` at 4-space indentation breaks on 2-space indentation, tabs, or mixed indentation. The patch silently inserts the value at the wrong nesting level, producing a syntactically valid YAML file that means something completely different.

2. **Service name assumptions.** The regex assumed the service was named `openclaw`. Users who renamed their service to `agent`, `claw`, or anything else got no fix applied — the pattern didn't match, the regex returned the original file unchanged, and `doctor --fix` reported success.

3. **Multi-document YAML.** Docker Compose files can contain multiple documents separated by `---`. A regex that finds `services:` matches the first occurrence, which may not be the right one.

4. **Comments and anchors.** YAML supports comments (`#`), anchors (`&`), and aliases (`*`). Regex has no concept of these — it can match inside comments, break anchor references, or silently drop aliases during a "find and replace."

5. **Quoting variations.** `user: "1000:1000"`, `user: '1000:1000'`, and `user: 1000:1000` are all valid YAML. A regex matching one form misses the others.

None of these failures produced errors. The regex returned modified (or unmodified) text, the file was written, and the fixer reported success. The compose file was either unchanged (regex didn't match) or corrupted (regex matched the wrong thing).

## Context

Regex YAML patching is everywhere in DevOps tooling. It's the default approach because it's easy to write, requires no dependencies, and works on the first test case. Searching GitHub for `sed -i` + `docker-compose.yml` returns thousands of results — CI scripts, deployment automation, configuration management, all using regex to modify YAML.

The fundamental problem: YAML is a context-sensitive format and regex is a context-free tool. You cannot correctly parse YAML with regular expressions any more than you can parse HTML with them. The difference is that YAML failures are silent — a malformed HTML page renders badly, but a malformed YAML file either parses to a different meaning or fails at runtime, long after the patching step.

This matters most in auto-remediation tools like `doctor --fix`. A manual edit has a human reviewing the result. An automated fix runs unattended and reports success. If the fix is wrong, the user trusts a broken config.

## The Fix

We replaced all regex-based compose patching with a proper YAML parser (`yaml` library — `parse()` and `stringify()`). The new `patchCompose()` function:

1. **Reads the file** and parses it as a YAML object
2. **Discovers services dynamically** — `getFirstServiceEntry()` reads `Object.keys(compose.services)` instead of hardcoding a service name
3. **Modifies the parsed object** — adding `cap_drop`, injecting `security_opt` entries, or setting `user` on the actual data structure
4. **Round-trip validates** — the modified object is stringified back to YAML and re-parsed to verify the output is valid
5. **Creates a backup** — writes a `.bak` file before overwriting the original
6. **Writes the result** — only after validation passes

The three fixers (`fixCapDrop`, `fixNoNewPrivileges`, `fixUserUid`) each take a parsed compose object, check whether the fix is needed, and mutate the service config if so. They don't know or care about YAML syntax, indentation, quoting, or service names.

```typescript
async function fixCapDrop(deployDir: string): Promise<FixResult> {
  return patchCompose(deployDir, "cap-drop", (compose) => {
    const entry = getFirstServiceEntry(compose);
    if (!entry) return null;
    const [, svc] = entry;
    if (Array.isArray(svc.cap_drop) && svc.cap_drop.includes("ALL")) return null;
    svc.cap_drop = ["ALL"];
    return { message: "Added cap_drop: [ALL] to compose" };
  });
}
```

The fixer operates on a JavaScript object. It has no opinion about whitespace, quoting, or document structure. The YAML library handles serialization.

## What We Learned

**Parse, don't pattern-match.** If a format has a parser, use it. The cost of adding a dependency is trivial compared to the cost of debugging silent corruption in production configs. This applies to YAML, JSON, TOML, XML, and any other structured format.

**Dynamic discovery beats hardcoded assumptions.** The regex assumed `openclaw` as the service name. The parser reads whatever service exists. This pattern — discover what's there instead of assuming what should be there — eliminates an entire class of "works on my machine" bugs.

**Round-trip validation catches what unit tests miss.** Parsing the output of your own serialization is a one-line check that catches encoding bugs, truncation, and format errors that no amount of pattern matching can detect.

**Auto-fix tools need higher correctness standards than manual tools.** When a human runs `sed`, they check the result. When automation runs a fixer, it reports success and moves on. Every automated modification should validate its own output.

---

## How ClawHQ Handles This

All compose file modifications in ClawHQ now use the YAML parse/modify/validate/backup/write pipeline. The `patchCompose()` helper in `src/operate/doctor/fix.ts` is the single path for all compose mutations. It creates a `.bak` backup before every write and validates round-trip integrity.

`clawhq doctor --fix` reports exactly what it changed and can be re-run safely — if a fix is already applied, the fixer detects it and skips.
